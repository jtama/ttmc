Quelles sont les étapes du scheduling framework ?

  1. Queueing / Admission : le pod arrive dans la file du scheduler.

Scheduling cycle

  1. PreFilter : prépare et valide des données partagées pour la phase de sélection.
  2. Filter : chaque nœud candidat est filtré selon des contraintes (affinity/anti‑affinity, taints/tolerations, ressources disponibles, node selectors, volumes attachables, etc.). On garde la liste des nœuds valides.
  3. PostFilter : intervient si aucun nœud n’a passé le Filter (ex. pour déclencher des actions d’échec ou de remédiation comme re‑préemption).
  4. Score : les nœuds valides sont notés (scoring) par les plugins ; les scores sont normalisés et additionnés pour obtenir le(s) meilleur(s) nœud(s).
  5. Reserve : réservation temporaire des ressources sur le(s) nœud(s) choisi(s).
  6. Permit : pause possible avant le binding : un plugin peut retarder l’ordonnancement (ex. attendre d’être approuvé par une politique, coordination multi‑cluster, ou pour la préemption). Peut aussi rejeter le pod.

Binding cycle

  1. PreBind : dernières vérifications / préparation juste avant la liaison.
  2. Bind : action qui lie le Pod au Node : appel à l’API kube‑apiserver pour créer l’objet binding (effectif « scheduled »).
  3. PostBind : notifications après le bind (ex. notifications, instrumentation, déclenchement de handlers externes).
